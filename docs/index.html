Intent Attention Kernel

Intent-Aware KV Execution for Agentic Long-Context Inference

"Attention should not pretend context is flat — and KV execution should not pretend every block is equally useful."

View GitHub Repo Read Architecture See Benchmarks

Why This Matters

Long-context inference is increasingly dominated by KV-cache capacity, memory bandwidth, page movement, and structured context management — not just attention compute.

Traditional attention asks: "How do we compute attention faster?"

Intent Attention Kernel asks: "What if the runtime tells the execution layer which KV regions matter?"

The missing interface is not just between model and kernel. It is between runtime intent and KV execution.

The Problem: Context Is Not Flat

Agentic workloads contain structurally different regions. A generic dense attention path treats all of them as one undifferentiated KV stream.

Sys Prompt Memory Summary Retrieved Doc 0 Retrieved Doc 1 Tool Output Scratchpad Recent Turns

Not all blocks deserve equal treatment. System prompts are always critical. Recent context is usually high priority. Retrieved documents may be relevant or irrelevant. Old tool output may be optional. Scratchpads may be compressible or skippable.

Core Idea: Semantic KV Block Metadata

Represent each semantic region as a block with an attention policy. The runtime can then select, score, quantize, and schedule KV blocks before they reach the attention kernel.

Python — Block Layout Definition
# Define semantic regions with policies
layout = BlockLayout([
    SemanticBlock("system_prompt",    0,     512,   BlockPolicy.ALWAYS),
    SemanticBlock("retrieved_doc_0", 512,   4096,  BlockPolicy.ATTEND, score=0.82),
    SemanticBlock("retrieved_doc_1", 4096,  8192,  BlockPolicy.SKIP,    score=0.21),
    SemanticBlock("recent_context",  8192,  12288, BlockPolicy.RECENT),
])

# Compute attention only over selected blocks
out, debug = semantic_block_attention(q, k, v, layout, return_debug=True)

Block Policies:

PolicySelectedDescription
ALWAYSYesAlways attend — system prompts, global context
ATTENDConditionalAttend based on a relevance score threshold
SKIPNoIgnore entirely — scratchpad, filler
RECENTYesAlways selected — sliding window / recent turns
GLOBALYesAlways selected — global memory tokens

Do not compute and then mask; expose structure early enough to avoid the work.

Five Pillars of the Project

The project explores five complementary mechanisms for exposing runtime intent to the KV execution layer.

Pillar 1

Semantic KV Block Selection

The runtime marks regions of context with policies. The CPU reference gathers selected K/V tokens and computes attention only over selected regions. This is the core interface between runtime intent and the attention layer.

Pillar 2

Selected-Block Attention Reference

A PyTorch CPU oracle proves correctness for non-causal selected-block attention. It compares selected-KV behavior against dense attention over the same selected K/V tensors. Causal selected-block attention is intentionally not implemented yet.

Pillar 3

Dynamic Block Scoring

A lightweight heuristic scores candidate blocks using query-to-block cosine similarity. This is not a trained router; it is a prototype control-plane signal for modeling the surface that a future runtime could consume.

Pillar 4

IntentQuant-KV

Not every KV block deserves the same precision. Critical regions can stay high precision (FP16/FP8) while lower-score or colder blocks may use INT8, INT4, INT4_RESIDUAL, or SKIP. No accuracy or perplexity preservation is claimed.

Pillar 5

Speculative KV Prefetch Simulation

Adjacent decode steps may reuse similar KV regions. A prefetcher can predict likely next-step pages. Current benchmark simulates hit rate only. Prefetch must never affect correctness. No real latency speedup is claimed.

IntentQuant-KV Deep Dive

Quantization should not be only a global model setting. It can also be an execution policy driven by runtime intent. Uniform KV quantization applies the same precision across different semantic regions; IntentQuant-KV explores block/page-level precision assignment based on policy, recency, score, and memory pressure.

Precision Ladder

FP16Critical blocks — system prompts, global memory, recent context — full precision
FP8Important selected blocks — high-scoring attended documents
INT8Relevant but non-critical blocks — moderate-score attended regions
INT4_RESIDUALMedium-score blocks — useful but lower confidence; residual correction
INT4Cold or low-score blocks — old tool outputs, low-relevance documents
SKIPSkipped blocks — zero KV bytes contributed

Sample Policy Assignment

BlockPolicyExample PrecisionReason
system_promptALWAYSFP16Critical instruction context; attended every step
recent_contextRECENTFP8 / INT8High reuse and recency
retrieved_doc_highATTENDINT8Relevant selected document
retrieved_doc_mediumATTENDINT4_RESIDUALUseful but lower confidence
old_tool_outputSKIPINT4 / SKIPCold or irrelevant

Limitations: No production quantization kernel. No model-quality or perplexity validation. Fake quantization only. Dequant overhead may dominate. Real benefit depends on bandwidth pressure, page reuse, hardware support, and attention fusion.

Benchmarks

All benchmarks run on CPU and do not require CUDA or Triton. CPU Ratio is not a GPU speedup claim.

CPU Ratio = dense_time / semantic_time on PyTorch's CPU backend. It is affected by PyTorch dispatch overhead, gather overhead, cache behavior, tensor size, and small-batch effects. CPU Ratio is not a GPU speedup prediction.

bench_cost_model.py

Analytical FLOP and KV-byte savings from selected-block attention. Zero-tensor arithmetic comparing dense vs selected-attention cost.

Does not prove: real GPU speedup

bench_cpu_reference.py

CPU timing sanity check for dense vs selected-block reference paths. Measures PyTorch overhead on CPU only.

Does not prove: fused kernel performance

bench_kv_quant.py

KV byte savings model for selected INT8-style KV pages. Compares fp16 dense storage vs int8+scale for selected pages.

Does not prove: real memory bandwidth reduction

bench_prefetch.py

Simulated next-step KV page prediction and hit-rate behavior for speculative prefetch during agentic decode.

Does not prove: real latency hiding

bench_dynamic_scoring.py

Synthetic query-to-block cosine-similarity scoring behavior across varying block counts.

Does not prove: routing quality or general solution

bench_intent_quant.py

IntentQuant-KV mixed-precision byte model and fake quant/dequant reconstruction metrics for FP16/FP8/INT8/INT4/INT4_RESIDUAL.

Does not prove: model quality or perplexity preservation

Causal Attention Limitation

Non-causal selected-block attention is implemented and verified in the CPU reference. Causal selected-block attention is intentionally not implemented because selected KV tokens retain original context positions. A correct implementation requires:

Requirements for Causal Selected-Block Attention

  • Explicit query_positions or per-query causal bounds
  • Query-position-aware masking in original coordinates
  • Partial-page token bounds for paged KV layouts
  • Logical KV ordering preservation
  • Page deduplication without reordering
  • Causal correctness for both prefill and decode phases

Calling semantic_block_attention(..., causal=True) currently raises NotImplementedError.

What Is Not Claimed

Honest Boundaries

  • No GPU speedups are claimed.
  • No production-ready Triton/CUDA kernel is claimed.
  • No real NVIDIA hardware validation has been performed.
  • Quantization has not been validated for model accuracy or perplexity.
  • No superiority over KIVI, KVQuant, or TurboQuant is claimed.
  • No production quantization kernel is provided.
  • No model quality guarantee is made.
  • Prefetch has not been validated for real latency improvement.
  • Dynamic scoring is a heuristic, not a trained routing model.
  • CPU Ratio is not a GPU speedup.
  • Analytical KV/FLOP savings are not measured GPU performance.

Future Roadmap

Phase 1

CPU-First Correctness

Metadata surface, block selection, dense baseline, selected-block reference, cost model, synthetic traces, benchmarks, test suite.

Phase 2

Query-Position-Aware Causal Attention

Implement correct causal selected-block attention with explicit query_positions or per-query causal bounds.

Phase 3

Paged KV Block Table

Support for partial-page token bounds and logical KV ordering preservation.

Phase 4

Triton Selected-Block Attention Kernel

Iterate only over physical pages from the block table; skip unused pages entirely.

Phase 5

IntentQuant-KV Dequant-in-Attention Fusion

Fuse per-page dequantization with the attention kernel to reduce memory round-trips.

Phase 6

Prefetch / Staging on Real GPU Hardware

Experiments with prefetch hints and latency hiding on NVIDIA GPUs.

Phase 7

Integration Experiments

vLLM and HuggingFace Transformers integration experiments for real inference workloads.
